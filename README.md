# Техническое задание — WebinarPulse

**Платформа аналитики вебинарных воронок**

- **Проект:** EcomSpace
- **Версия:** 1.0
- **Дата:** 11 марта 2026
- **Автор:** Даниил

---

## 1. Общее описание проекта

### 1.1. Цель

Создать узкоспециализированную real-time платформу для аналитики вебинарных воронок EcomSpace. Платформа должна показывать полную картину по каждому конкретному вебинару: сколько человек зарегистрировалось, сколько пришло, сколько сделало заказ, сколько оплатило — с детализацией по UTM-меткам и произвольным меткам из GetCourse.

### 1.2. Ключевая проблема

Вебинары проходят дважды в неделю (вторник и четверг). Люди могут приходить повторно. Стандартная аналитика GetCourse не привязывает заказы к конкретному вебинару — человек мог посетить вебинар год назад, а купить сегодня, и такой заказ некорректно приписывать текущему вебинару.

### 1.3. Решение

Система атрибуции с **окном 72 часа** после последнего участия. Если человек посетил вебинар и оплатил в течение 72 часов — заказ привязывается к этому вебинару. Если позже — заказ попадает в категорию «отложенные оплаты» (видны отдельно, не искажают статистику конкретного веба). Окно атрибуции настраивается в интерфейсе.

> **Расписание вебинаров и окна регистраций:**
>
> - Веб 1 (вторник): регистрации с 20:00 четверга по 20:00 вторника
> - Веб 2 (четверг): регистрации с 20:00 вторника по 20:00 четверга
> - Часовой пояс: Москва (UTC+3)

## 2. Архитектура

### 2.1. Схема потока данных

```
GetCourse (событие) → Webhook → n8n (обработка + валидация) → API бэкенда → PostgreSQL → Фронтенд (дашборд)
```

### 2.2. Стек технологий

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| Бэкенд | Node.js + Express / Fastify | Быстрый старт, легко писать через Codex |
| БД | PostgreSQL | Надёжно, хорошо с временными данными |
| ORM | Prisma | Типизация, миграции, автогенерация |
| Фронтенд | React + Vite + Tailwind | Быстрый UI, графики через Recharts |
| Графики | Recharts | React-native, простой API |
| Оркестрация | n8n (self-hosted) | Уже используется в стеке |
| Хостинг | VPS | Полный контроль, всё в одном месте |
| Авторизация | Простой логин/пароль + JWT | 2–5 пользователей, не нужна сложная система |

### 2.3. Структура проекта (monorepo)

```
webinar-pulse/
├── packages/api/       — Бэкенд (Express/Fastify + Prisma)
```

## 3. Модель данных (Prisma-схема)

### 3.1. Таблица `webinars`

Каждый вебинар — это конкретная дата. Система автоматически создаёт вебинар, когда приходит первая регистрация на новую дату.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `scheduled_at` | DateTime | Дата и время начала вебинара (напр. 2026-03-10T20:00 MSK) |
| `reg_window_start` | DateTime | Начало окна регистрации (20:00 предыдущего веба) |
| `reg_window_end` | DateTime | Конец окна регистрации (20:00 в день веба) |
| `day_of_week` | Enum | `TUESDAY` \| `THURSDAY` |
| `status` | Enum | `UPCOMING` \| `LIVE` \| `COMPLETED` |
| `created_at` | DateTime | Автосоздание |

### 3.2. Таблица `contacts`

Уникальный контакт = уникальный email или `gc_user_id` из GetCourse.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `gc_user_id` | String? | ID пользователя в GetCourse (основной ключ матчинга) |
| `email` | String? | Email (резервный ключ матчинга) |
| `phone` | String? | Телефон (дополнительный ключ) |
| `name` | String? | Имя контакта |
| `first_seen_at` | DateTime | Когда впервые появился в системе |

### 3.3. Таблица `registrations`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `contact_id` | FK → contacts | Кто зарегистрировался |
| `webinar_id` | FK → webinars | На какой веб (определяется по окну регистрации) |
| `gc_deal_id` | String | ID сделки в GetCourse |
| `utm_source` | String? | UTM-метка |
| `utm_medium` | String? | UTM-метка |
| `utm_campaign` | String? | UTM-метка |
| `utm_content` | String? | UTM-метка |
| `utm_term` | String? | UTM-метка |
| `custom_labels` | JSON | Произвольные метки из GetCourse (ключ-значение) |
| `registered_at` | DateTime | Дата/время регистрации |

### 3.4. Таблица `attendances`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `contact_id` | FK → contacts | Кто пришёл |
| `webinar_id` | FK → webinars | На какой веб (определяется по дате веба) |
| `gc_deal_id` | String | ID сделки в GetCourse |
| `attended_at` | DateTime | Когда подключился |

### 3.5. Таблица `orders`

Это ключевая таблица — именно здесь работает логика атрибуции.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `contact_id` | FK → contacts | Кто заказал |
| `attributed_webinar_id` | FK → webinars? | NULL = отложенная оплата (вне окна 72ч) |
| `last_attendance_id` | FK → attendances? | Последнее посещение перед заказом |
| `gc_deal_id` | String | ID сделки в GetCourse |
| `status` | Enum | `NEW` \| `PAID` \| `CANCELLED` |
| `amount` | Decimal? | Сумма заказа |
| `product_name` | String? | Название продукта |
| `attribution_type` | Enum | `DIRECT` \| `DEFERRED` \| `UNATTRIBUTED` |
| `ordered_at` | DateTime | Дата создания заказа |
| `paid_at` | DateTime? | Дата оплаты (NULL = не оплачен) |

### 3.6. Таблица `webhook_log`

Лог всех входящих вебхуков для дебага и повторной обработки.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ |
| `event_type` | String | `registration` \| `attendance` \| `order` \| `payment` |
| `payload` | JSON | Полное тело вебхука ас-из |
| `processed` | Boolean | Успешно обработан? |
| `error` | String? | Текст ошибки если была |
| `received_at` | DateTime | Когда пришёл |

## 4. Алгоритм атрибуции заказов

**Это ядро системы.** Когда приходит заказ или оплата, система выполняет следующую логику:

1. Найти контакт по `gc_user_id` или `email`
2. Найти последнее участие (`attendance`) этого контакта
3. Вычислить разницу: `дата_заказа − дата_последнего_участия`
4. Если разница ≤ 72 часа → `attributed_webinar_id` = вебинар последнего участия, `attribution_type` = `DIRECT`
5. Если разница > 72 часа → `attributed_webinar_id` = NULL, `attribution_type` = `DEFERRED`
6. Если участие не найдено вообще → `attribution_type` = `UNATTRIBUTED`

> **Обработка оплаты (два сценария):**
>
> - **Сценарий A:** Вебхук со сменой этапа сделки → находим существующий order по `gc_deal_id`, обновляем `status` → `PAID`, заполняем `paid_at`
> - **Сценарий B:** Новая сделка-оплата → матчим по `email`/`gc_user_id` с существующим order (статус `NEW`), обновляем. Если не нашли — создаём новый order со статусом `PAID`

## 5. API эндпоинты

### 5.1. Webhook-эндпоинты (приём данных из n8n)

| Метод | URL | Описание |
|-------|-----|----------|
| POST | `/api/webhooks/registration` | Новая регистрация на вебинар |
| POST | `/api/webhooks/attendance` | Участие в вебинаре |
| POST | `/api/webhooks/order` | Новый заказ (запускает атрибуцию) |
| POST | `/api/webhooks/payment` | Оплата (смена этапа или новая сделка) |

### 5.2. Данные для дашборда

| Метод | URL | Описание |
|-------|-----|----------|
| GET | `/api/webinars` | Список вебинаров + сводные цифры |
| GET | `/api/webinars/:id` | Детальная статистика по вебу |
| GET | `/api/webinars/:id/funnel` | Воронка: реги → пришли → заказы → оплаты |
| GET | `/api/webinars/:id/by-source` | Разбивка по UTM-меткам |
| GET | `/api/analytics/overview` | Общая сводка по всем вебам |

## 7. Фронтенд — Дашборд

### 7.1. Главная страница (Overview)

- Карточки-сводки: всего регистраций, участий, заказов, оплат за период
- График динамики по вебам (бар-чарт, каждый бар = 1 вебинар)
- Таблица последних вебинаров с ключевыми метриками
- Фильтр по датам

### 7.2. Страница вебинара (Webinar Detail)

- Воронка-визуализация: Регистрации → Участия → Заказы → Оплаты (с % конверсии между этапами)
- Разбивка по UTM-источникам: таблица с колонками source/medium/campaign + реги/пришли/заказы/оплаты
- Таблица участников: имя, email, метка, пришёл?, заказал?, оплатил?
- Сравнение с предыдущим вебом (дельта по каждому этапу)

### 7.3. Страница отложенных оплат

Список заказов с `attribution_type` = `DEFERRED` или `UNATTRIBUTED`. Позволяет видеть покупки, которые не привязаны к конкретному вебу, но являются частью общей выручки. Показывает: когда был последний вебинар человека, сколько прошло дней до оплаты.

### 7.4. Настройки

- Окно атрибуции (по умолчанию 72 часа, настраивается)
- Расписание вебинаров (дни недели + время)
- Управление пользователями (логин/пароль, до 5 аккаунтов)
- API-ключ для webhook-эндпоинтов (базовая безопасность)

## 8. Ключевые метрики дашборда

| Метрика | Формула | Где показывается |
|---------|---------|-----------------|
| Регистрации | `COUNT(registrations)` | Overview + Detail |
| Участия | `COUNT(attendances)` | Overview + Detail |
| Доходимость | `участия / регистрации` | Detail |
| Заказы (DIRECT) | orders с атрибуцией | Overview + Detail |
| Оплаты (DIRECT) | orders (PAID + DIRECT) | Overview + Detail |
| CR веба | `оплаты / участия` | Detail |
| Выручка веба | `SUM(amount)` PAID+DIRECT | Detail |
| Отложенные оплаты | orders (DEFERRED) | Deferred страница |

## 9. n8n воркфлоу

Необходимо создать 4 воркфлоу в n8n (или 1 с роутингом):

1. **Регистрация:** GC Webhook → Маппинг полей (email, utm, метки) → POST `/api/webhooks/registration`
2. **Участие:** GC Webhook → Маппинг (gc_user_id, email) → POST `/api/webhooks/attendance`
3. **Заказ:** GC Webhook → Маппинг (gc_user_id, email, amount, product) → POST `/api/webhooks/order`
4. **Оплата:** GC Webhook → Определить тип (смена этапа / новая сделка) → POST `/api/webhooks/payment`

> **Валидация в n8n:**
>
> - Проверять наличие обязательных полей (`gc_user_id` или `email`)
> - Логировать ошибки в отдельный нод (для дебага)
> - Добавить API-ключ в заголовок `Authorization`

## 10. Безопасность

- Webhook-эндпоинты защищены API-ключом в заголовке (`X-API-Key` или `Authorization: Bearer`)
- Дашборд за логином/паролем + JWT-токены
- Rate limiting на webhook-эндпоинты (защита от спама)
- HTTPS обязателен (Let's Encrypt + nginx reverse proxy)
- Идемпотентность: повторный вебхук с тем же `gc_deal_id` не создаёт дубликат

## 13. Инструкции для разработки (для Codex / Claude)

При разработке платформы следуйте этим принципам:

- **Язык:** TypeScript везде (бэк + фронт). Strict mode.
- **Все даты** хранить в UTC, отображать в MSK (UTC+3) на фронте.
- **Окно регистрации:** 20:00 MSK предыдущего веба — 20:00 MSK текущего веба.
- **Матчинг контактов:** сначала по `gc_user_id`, затем по `email`, затем по `phone`. Если не найден — создать новый.
- **Идемпотентность:** дубликат вебхука с тем же `gc_deal_id` должен обновлять, а не дублировать запись.
- **webhook_log:** логировать КАЖДЫЙ входящий вебхук до обработки.
- **Окно атрибуции** (72ч) должно быть конфигурируемым через таблицу `settings`, не хардкодить.
- **Фронтенд:** минималистичный, чистый UI. Tailwind + Recharts. Без лишних украшений.
- **API-ключ** для вебхуков хранить в `.env`, проверять через middleware.
